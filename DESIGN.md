# Hanoba — DESIGN

> 植物写真の SNS。1投稿 ＝ 正方形写真1〜4枚 ＋ ひとこと（必須）。**「育てて、見せる。」**
> **mypace（Nostr）のスキン**であり、自前のバックエンドを持たない。

Instagram より早く・かんたんに、文章を添えて植物を見せる場。世界観は**暗くシックな大人のコレクター部屋**（§5.0）。畑・ビカクシダ・種まき・パキポ・アガベの自慢と、実験の逐次メモを兼ねる。長期目標は **架空の植物好き都市ハノーバ(Hanoba)をみんなで作る**こと。

---

## 1. 原則（規律）

1. **正方形固定** — 投稿画像は正方形に矯正。クロップはドラッグで位置決め。1投稿に1〜4枚まで入れられる。
2. **一言必須** — 写真だけの投稿を許さない。`content` 空は弾く。これが心臓部。
3. **動画非対応** — 身バレ（背景の映り込み・環境音・反射・複数フレーム）回避。静止画のみ。
4. **植物だけ** — テーマは固いが、その枠内は何でも OK（自慢・実験メモ）。

## 2. アーキテクチャ

- **自前バックエンドなし。** 状態は全部 Nostr に乗る。投稿は **mypace 経由でなく共有 Nostr 空間へ直接**行う。「mypace のスキン」の実体は **同じ Nostr タグ規約を使うこと**。
  - 補足（#28・実装確認）: mypace 自体は Workers＋D1 の API を持つが、**Hanoba はそれを使わない**。投稿（kind:1）・削除（kind:5）・プロフィール（kind:0）・画像アップロード/削除（nostr.build NIP-96）は**全てクライアント側**で完結し、バックエンド不要を維持する。
- **アカウント（#28）**: nsec はブラウザ生成・localStorage 保存（`getOrCreateSecretKey`）。**ユーザー名（kind:0）を入れたら投稿できる**（見るだけ→参加）。**既存ユーザーは nsec を持ち込める**（`AccountName` の「すでにアカウントをお持ちですか？」→ `importNsec` → 既存 kind:0 全体（名前＋picture/about/websites）を `fetchMyProfileResilient` で引き継いで控えにシードし reload）。取得は bounded retry で行う（単発だと接続直後に websites を載せた版を取りこぼし、控えに `websites:[]` を焼いてしまう＝#93）。mypace 等のユーザーがアカウントを無駄に増やさない。鍵は端末（localStorage）にのみ保存。
- **削除＝写真と一蓮托生（#28）**: 投稿削除は ①NIP-09 `kind:5` を publish（mypace/Nostr 上で不可視化）＋②**nostr.build の画像を本人の鍵で NIP-96 削除**（`DELETE /api/v2/nip96/upload/<sha256>`・NIP-98 認証）。「自分の植物」`/me` から行う。D1 を持つ mypace と同じ"消せる一芸"を backendless で再現（画像 URL は投稿本文に内包されるので upload 履歴 DB も不要）。鍵を失うと過去投稿は消せない＝鍵の重要性に注意。
- **編集＝確認つき再投稿（#300・mypace 由来）**: Nostr はイベント不変なので「編集」＝**新しい投稿を publish して旧イベントを NIP-09 `kind:5` で削除**する（`client.editPost`＝**新規 publish → 旧削除**の順。先に消さない＝publish 失敗で投稿が消える事故を防ぐ）。**写真は本文 URL を再利用する（再アップロードしない）**ので、`deletePost` と違い nostr.build の実体は消さない（消すと再投稿側の URL が死ぬ）。**いいね・コメントは旧イベント id に紐づくため引き継がれない**＝`/me`（`MyGrid`）の編集ボタン→`EditPost` モーダルで本文（画像 URL を除いた全文＝`#`タグも本文の一部として直す）を直し、保存時に**「いいね N・コメント M は引き継がれません」確認を1段挟んで**から実行する。写真の追加/削除/差し替えは現状なし（本文編集のみ・follow-up）。
- **市民の活動スタッツ（#272・脱ゲーム化）**: `/me`（`MyGrid`）に**活動スタッツ節**（`CitizenStats`）を出す。**全て t:hanoba 投稿からのクライアント集計＝backendless・新たな身バレ無し**（公開投稿を数えるだけ・ステラ/閲覧数は不採用）。項目＝**投稿数・写真枚数（imageUrls 合計）・育てた品種数＋一覧・居住日数（最古投稿から・"在籍"でなく"居住"）**。品種の同定は `tallyVarieties`（`buildFuda` 一本化）を再利用＝ランキングと一致。**育てた品種一覧は横ページング（#388・`VarietyPager`）**＝品種が 1000 種あっても**全件チップを縦に積まない**（巨大 DOM・長大スクロールでプロフィールが破綻し、下の節〔緑の総面積／撮影の草〕に辿り着けなくなるのを防ぐ）。**1 ページ 10 件**にチャンクして**現在ページだけ描画**＝節の高さが常に有界。手帳のように横へめくる（**←→ボタン・キーボード矢印・タッチスワイプ**＋**ぼかしページ遷移**）＝市民手帳（`CityHallBook`）と同じ作法で `carousel.ts` の純関数（`swipeDirection`/`swipeProgress`/`swipeToBlur`）を**共有**する。**端はクランプ（非 wrap）**＝先頭で←・末尾で→は no-op（ボタン disabled）。めくりは `key={page}` の中身だけ blur（枠は固定）し **reduced-motion／1 ページなら blur 無効**。**10 件以下はページャ UI（←→・インジケータ）を出さず従来どおり並べるだけ＝退行なし**。チャンクは純関数 `src/lib/feed/varietyPager.ts`（`chunk`/`pageCount`/`clampPage`・`PAGE_SIZE=10`・テスト済み）。チップの見た目（glass・学名 `SciName`＋和名＋×N・`discoverTagsHref` の discover 絞り込みリンク）は不変。集計は純関数 `computeCitizenStats(posts, catalog, hasName, now)`（`src/lib/feed/stats.ts`・`now` を引数で受け決定的）。**市民レベルは 旅人(L0=名乗り前) → 市民(L1) → 市民L2 → …市民Ln**（`citizenLevelLabel`・#272 kako-jun「古参/訪問者という語は使わない・名乗ったら市民のままレベルが上がる」。L2 は名乗り＋5投稿＋居住14日〜＝既存 `citizenLevel`・Ln L3-L6 は複合〔居住×投稿〕で進む）。`CitizenStats` は自分/他人共通（**他人の公開プロフ `/u?npub=`・実装済**）で、catalog は動的 import（PostGrid 同型）。**称号/実績バッジは脱ゲーム化で撤去（#272・市長は王様でなく授ける権限はない／未解除 ??? が面積を取って煽るのは hanoba の静けさに合わない）**＝レベルは塗りピル（CTA 見え）でなく**うっすい囲みの静かなステータスラベル**で残すだけ、表示は**事実のスタッツのみ**（投稿/写真/品種/居住＋育てた品種一覧）。称号の置換＝**「緑の総面積」（#310/#344/#387・`GreenArea`）**＝写真の**緑画素割合**を積み「今まで街に足した緑の累計」を**小ラベル「緑100%の写真換算」＋大きく右寄せの値「約○○枚分」**で静かに出す（#439 kako-jun・値を目立たせる＝ラベルと値を左右に分け値を `text-2xl` 右寄せ／i18n は `green.cumulative.label`＋`green.cumulative.value`）。エゴの称号でなく貢献。#344 で**草グリッド・濃淡凡例・先頭写真限定・直近60件上限は廃止**したが、#344 の「全投稿の全写真を素直に累計」は**全数を一斉に DL/デコードして重い**ため、#387 で kako-jun 合意のもと**代表サンプル（上限 N 枚を均等抽出）からの外挿で概算**に変更（1000 枚でも一定コスト・厳密でなくてよい）。画素読みは `crossOrigin=anonymous` の canvas（hanoba は nostr.build=CORS 反射ホストにアップロード＝読める）・他クライアント由来の非CORS画像は tainted で `getImageData` が throw するので try/catch でスキップ（概算・グレースフル）。純粋判定/抽出は `src/lib/feed/green.ts`（`greenRatio`＝緑優勢画素〔g>40 && g>r+8 && g>b+8〕の割合／`cumulativeGreen`＝読めた写真の緑割合の総和＝換算枚数＋読めた枚数／`pickSampleIndices`＝全数→上限 cap の均等抽出インデックス／`estimateCumulativeGreen`＝代表サンプルから全数への外挿概算〔抽出時 sampled:true〕・テスト済み）。画素読みの副作用は `GreenArea.tsx`（上限 cap=120・同時実行 6・URL 単位のセッション内キャッシュ・#387）。**撮影の草（#272 段階4・`ActivityHeatmap`）**＝**いつ・どれだけ撮ったか**を GitHub 風の日別ヒートマップ（週列×7曜日・1マス=1日・濃淡=その日の写真数〔鼓門 JST 暦日〕・**4段階濃淡〔なし/少(1枚)/中(2枚)/多(3枚〜)・#440 kako-jun。0=なしは薄く、緑3段は明瞭に・少が0と紛れない〕**・直近12週・**左に曜日ラベル（#345・**3文字略 `Sun`〜`Sat`・全7行を省略せず固定**〔#345 の「全言語英語・曖昧なし」の意図維持・行=0日〜6土〕）で縦軸を示し、濃淡凡例（少→多の緑3段のみ・「なし」は出さない）は見出し行の右に置く（#440）**・見出し（撮影の草）と「直近N週」は上下中央に揃える（#440）・各週列に7行を貫く控えめな縦トラックを敷き暦週そろえの空白角を「列の中のまだ無い日」と読ませる〔#389〕・kako-jun「縦横わからない/ずれて見える」）＋**連続記録**（現在の連続/最長連続・今日未投稿でも昨日まで続けば生存）で静かに出す。**撮影日基準（#324・kako-jun「毎日撮って週末まとめ投稿する人が報われるべき」）**＝各写真の撮影日（`shot_date` タグ・per-photo）で数え、撮影日が無い写真/投稿は created_at 日にフォールバック（`postActiveDays`）。緑の総面積（緑の貢献量・**全写真からの概算換算枚数**・#344/#387）とは**別軸＝撮影の頻度/継続（per-day）**で補完的。純関数 `src/lib/feed/activity.ts`（`jstDayIndex`/`ymdToJstDayIndex`/`postActiveDays`/`activityHeatmap`/`streaks`/`activityLevel`・テスト済み）。バッジ/手帳連動演出は脱ゲーム化で不採用（事実の可視化だけ）。
- **フロントエンドのみ**: Astro（静的生成）＋ React islands ＋ **PWA**（`@vite-pwa/astro`）。**Cloudflare Pages** にデプロイ（Workers も D1 も不要）。
- クライアントが自分で行う:
  - **鍵** — ブラウザ生成・localStorage 保存・署名（NIP-07 拡張も可）。
  - **画像アップロード** — nostr.build へ直接（NIP-98 認証）。**EXIF はサーバ側で自動削除**（身バレ＝自宅 GPS 漏れ対策が無料で効く）。維持費ほぼゼロ。
  - **投稿・購読** — Nostr リレーへ直接 publish / subscribe。
- **mypace から見える鍵＝タグ**: 投稿に `['t','mypace']` を必ず付ける（mypace タイムライン出現の条件）。詳細は §6。
- **mypace の記録 API への ping は後回し（任意・優先度低）**: ping すれば mypace の通知・スコア（ステラ/通し番号）系にも乗るが、無くても `t:mypace` だけで mypace タイムラインには出る。必要になったら将来。
- **断り書きを常設**: 「Nostr フィードを表示しているだけ・二重投稿でもコピーでもない・mypace でも見られます」。
- → mypace の正式リリースを待たず独立公開可。むしろ hanoba が mypace を賑わす燃料になる。

## 3. 実装方針

- 既存実績のある **Astro ＋ React islands ＋ PWA（`@vite-pwa/astro`）＋ Cloudflare Pages ＋ Tailwind** 構成を骨格として踏襲する（独自バックエンド／DB 層は持たない）。
- **mypace の既存コンポーザー（アップロード／クロップ／レトロ加工）を流用**する。React 同士なので島として移植する。
  - クロップ → **正方形ロック＋ドラッグで位置決め**に変更。**角度回転（#314・mypace ImageEditor 方式）**: `CropFrame` 下のコントロール＝**[左90°][−0.5°][===微調整スライダ===][+0.5°][右90°]**（90度ボタン＝向き直し／スライダ＋0.5刻みボタン＝水平出し・±15°）。**プレビューは即時の CSS `transform: rotate()`**（blob 再生成の遅延なし＝kako-jun「遅れて回転する／何も起きない」を解消）。クロップ枠は元画像の box（軸整列）に引かれ、**焼き込みは `renderInPlaceRotation`**（mypace 同方式＝元画像と同寸の canvas に中心回転で描く・crop 矩形は元画像の自然座標のまま一致）で**プレビューと同じ見え**を再現する。**90/270 回転後のクロップ枠制限（#348）**: in-place 回転だと写真は中心の `min(W,H)` 正方形だけが見え左右（縦長は上下）は letterbox の空き帯になるが、react-image-crop の操作領域は回転前 box のままなので枠を空き帯（写真外）へ出せてしまう。純関数 `clampCropToVisible(crop, rotation, boxW, boxH)`（テスト済み）で**見えている中心正方形に clamp**する（ドラッグ中=`onChange`／90度跨ぎ=`quarter` 変化の effect で収め直し＋commit）。0/180・微調整回転は素通し（退行なし）、焼き込みは座標系不変なので**プレビューと一致**を保つ。crop は座標系が変わらないので**リセットしない**（v1 のリセットが招いた「未設定」表示も解消）。rotation は draft に永続化。純粋部 `rotationFine`（微調整成分）はテスト。在りし v1（回転済み blob を src 差し替え）は遅延と未設定の原因だったので撤回。**撮影日（#324）**: 添付時に各写真の EXIF `DateTimeOriginal`／ファイル名から撮影日（`YYYY-MM-DD`）を検出し（`src/lib/image/exif.ts`・純関数）、コンポーザの「撮影日」欄で**編集・除外**できる（不正確な EXIF 対策）。投稿時に distinct を kind:1 の `["shot_date", 日付]` タグへ載せる（本文は汚さない・他クライアントは無視・`DraftImage.shotDate` は draft 永続化）。これで投稿日とズレても**撮影の草は撮影日で積まれる**（後日まとめ投稿でも撮った日数が点く）。
  - 植物写真向けの穏やかな補正 → ランダム適用でなく**選択式**。細かなスライダーは畑で使いにくいので増やさず、葉色や花色を潰さない控えめなプリセットにする。
    - **重ねがけ式プリセット**（#140/#155/#156）: 淡陽(明るさ)／翠露(S字トーン)／土香(逆S字トーン)／美華(彩度・彩度を上げるのは美華だけ)／影暮(周辺減光)／霞幻(周辺ぼかし)／線明(シャープ)。翠露＋土香 は相反するので同時選択で完全相殺。
    - **各効果は なし/弱/中/強 の4段**（#171）: プリセットは弱/中/強の3段（`levels`）を明示的に持ち、**中＝既定の効き**・線明（シャープ）は効きすぎたので中を控えめにする。チップは1回タップごとに なし→弱→中→強→なし で巡回し、**横長チップを3段階に塗りつぶして**現強度を見せる（色プリセットは持たず全部が効果トグル）。
    - **投稿時に canvas へ焼き込む**（プレビューと同じ見えを 1:1 出力に確定）。トーンカーブ(S字/逆S字)・周辺ぼかし(霞幻)・シャープ(線明)は CSS では表現しきれないため canvas のピクセル処理（LUT・blur 合成・畳み込み）で焼く。**プレビューは CSS（contrast/blur/減光オーバーレイ等）で近似**する＝畑で迷わない速さを優先し、最終の厳密さは焼き込み側に置く。実装は `src/lib/image/{presets,crop}.ts`。
    - **出力は長辺 1440px 上限に縮小（拡大はしない）・JPEG quality 0.85・フィルタは縮小後の小 canvas に焼く・複数枚は並列アップロード**（#167。nostr.build がどうせ縮小配信するため、巨大 JPEG の生成・CPU 処理・POST を短縮する。1:1 出力は不変）。
  - **下書きの自動保存・復元（#228）**: 書きかけのひとこと・写真・クロップ枠・フィルタ・並び順・選択中の写真を **IndexedDB に黙って自動保存し、開いたら黙って復元**する（「バックアップ／復旧」概念はユーザーに見せない・バナーも通知も出さない）。アップロード下書きは不変 Def でなく**状態側**なので `src/lib/composer/draft.ts`（SSR 安全・ランタイム依存ゼロのネイティブ IndexedDB 薄ラッパ）に分離し、写真 blob とメタ（本文/クロップ/フィルタ/並び/選択 id）を別ストアに持つ。本文が消えるのは ①本文右端の × ②自分で本文を空にする ③投稿成功 の3経路だけ＝**写真を選び直しても本文は保持**する（写真操作は本文に触れない）。
  - **写真の並べ替え（#274）**: 添付した複数写真（2枚以上）は、選択中の写真を **◀▶ で左/右へ1つずつ動かして順番を変えられる**（先頭＝カバー）。タップ＝編集選択は従来どおり。crop/filters は各写真（`DraftImage`）に内包されるので並べ替えは**配列入れ替えだけ**＝加工も写真と一緒に動く。並べ替えは純関数 `src/lib/composer/reorder.ts`（`moveById(items, id, delta)`・範囲外/不在は no-op・非破壊）に切り出し、Composer は `setImages` 経由で適用＝下書きの blobs(`order`)/meta(順序) effect が発火し並べ替え後の順序も自動保存・復元される。サムネ列直下に並べ替え行（`N枚目 / 全M枚`・端でボタン無効）。**◀ と ▶ はタップしやすいよう両端に離す**（中央にカウンタ・各ボタン 44px 標的）。
  - **画像編集の1手アンドゥ（#363/#393/#403/#406）**: 投稿前に、直前の画像編集を1手戻すボタンを**編集中写真（`CropFrame`）の右下隅にオーバーレイ**で置く（#406・撮影日=左下・枚数=右上のバッジと同じ流儀＝写真上の半透明ピル。`CropFrame` の `photoOverlayBottomRight` slot に渡す。親 slot は `pointer-events-none`、ボタンに `pointer-events-auto` でクロップのドラッグ操作と競合させない）。対象は kako-jun 指定の **角度（rotation）・フィルタ（filters）・Exif撮影日（shotDate/shotDateAuto）** と、**ユーザードラッグ由来のクロップ（crop）**（#393）。**クロップは自動センタークロップ・90度clamp 由来の commit は対象外**（画像ロード時の初期 commit や回転の副作用で履歴が汚れる/勝手にアンドゥが有効化するのを防ぐ。回転自体は別途 `onRotate` がアンドゥ対象）。**本文（一言）・タグは対象外**（本文はブラウザ標準のテキスト undo に委ねる）。配線は `CropFrame` の `onCropComplete(crop, fromUser)` で、ユーザーの `ReactCrop` `onComplete` だけ `fromUser=true`、`onImageLoad`・quarter clamp effect は `fromUser=false`。実装は `Composer` の `updateCurrentImage(patch, { userCrop, continuousRotation })` で変更前の `images` を履歴に積む ephemeral スタック（下書き永続化はしない）。**畳み込みは連続入力＝回転微調整スライダの1ドラッグだけを1手に潰す（ジェスチャ終端でリセット）**（#403。判定は**フィールド名でなく明示フラグ `continuousRotation`** で行う＝`CropFrame` のスライダ `onChange` だけ `setFine(v, true)` で連続を立て、`lastEditTagRef`＝畳んだ run の最初の状態だけ残す。さらにスライダの `onPointerUp`/`onTouchEnd`/`onMouseUp`/`onKeyUp` で `onRotateGestureEnd`＝`lastEditTagRef=null` を呼び、**1ドラッグ終端で畳み込みをリセット**するので次のドラッグは別の1手になる）。**離散編集＝90°・±0.5 ボタン・フィルタ・クロップ・撮影日は1アクション=1手＝毎回 snapshot を積み畳まない**（#403。旧挙動は「rotation 単独」というフィールド名で連続判定していたため 90°/±0.5 ボタンの離散クリックまで連続扱いで畳み、複数回の回転が1手にまとまっていた＝kako-jun の不満を直す。フィルタも on→on→off と続けて変えると、従来は同一 field 連続変更として畳み唯一の snapshot が「最初=空」になり off→undo で空のまま無変化だった実機バグを直す＝離散を畳まないことで off 直前の snapshot を積み戻せる）。**crop は `cropSyncToken` で undo 時に表示を再同期する**（#403。`CropFrame` は crop を内部 state＝uncontrolled で持ち `initialCrop` は `onImageLoad` の初期値専用なので、id 不変のままの undo は再マウントされず矩形が動かない。`undoLastEdit` が `setCropSyncToken((n)=>n+1)` で increment し、`CropFrame` の effect が外部復元値＋`imgRef` の自然寸法から `squareRectToPercentCrop`（`onImageLoad` と共通の純ヘルパ）で % crop を引き直して `setCrop` し直す。初回 token=0・未ロード時はスキップ＝`onImageLoad` が初期化を担う）。画像の追加/削除/並べ替えなど構造変更が入ったら履歴はクリア（古い blob 参照を復元しない）。履歴が空ならボタンは不活性。
- **一言の `#` ハッシュタグ補完**: 入力中の `#` を**過去に使われたタグから補完**し、同じ植物のタグへ投稿が自然に集積する（emergent taxonomy＝固定の分類表を持たずタグが育つ）。補完候補の源はリレー上の過去 `t` タグ。よく使う種は Wikipedia 由来で seed 可、細かい品種名（cultivar）は Wikipedia に無いので freeform `#`（過去使用のみで補完。エンティティ紐付け `@@` の UX を似せるだけ）。既存のメンション補完 UI を流用。

  - **方針確定（Twitter 型・en 基本・kako-jun session692）**: hanoba の実コンテンツ（投稿）は FeedGrid が**クライアントで Nostr から描画**＝静的 HTML に載らないので、言語別ルーティング（`/en/` `/ja/`・hreflang・言語別sitemap）で得る SEO は薄い殻（ヒーロー/about）だけで**ごく小さい**。hanoba は「翻訳マーケサイト」でなく **Twitter 型の SNS**（UGC は単一言語のまま・大手 SNS も言語別 URL を切らない）。よって:

## 4. 主要画面

- **投稿**: 撮影/選択（最大4枚）→ 写真ごとに正方形クロップ（ドラッグ）→ 植物写真向け補正（選択式）→ 一言（必須）→ 投稿
  - **撮影とアルバムはボタンを分ける（#29）**。単一 input に `capture="environment"` を付けるとスマホでカメラ直起動が優先されアルバムから選べなくなるため、`capture` 付き input（撮影）と `capture` 無し input（アルバム）を別ボタンで持つ。
  - **必須・不足理由の a11y（#151）**: 「一言（必須）」は視覚ラベルに「（必須）」と書かず（#149）プログラム属性で支援技術に伝える＝一言 `<textarea>` に `aria-required="true"`。投稿できない理由「あと ◯◯ を入れると投稿できます」は `role="status"`＋`aria-live="polite"` で状態変化を能動読み上げ。送信ボタンは不足があるときだけ `aria-describedby` でその不足理由を指し（充足したら属性ごと外す＝dangling 参照を作らない）、disabled の理由をボタンから辿れるようにする。
- **フィード（読めるフィード・#34/#50）**: hanoba の売り＝**本文を切らず読め、1クリックも要らない**（vs Instagram）。縦並びカード（PostCard）。sm 以上＝写真左／本文中／タグ右の縦列、モバイル＝写真上／本文／タグ下。**本文テキストからは #タグ を除き**（stripHashtags）タグは右列に出す。**デスクトップはカード高さを写真の正方形（sm:h-56）に固定**し、本文・タグをその中に収める（はみ出しは clip）＝写真の下に隙間を作らない。本文かタグが clip された時だけ**「続きを読む」**（フェードは使わない）でカード全体を展開し全文＋全タグを表示。タグが多くても**1列のまま下を見切る**（2列に折り返さない・#54）。**カードの非インタラクティブ領域はどこを押しても拡大**（PostDetail モーダル・写真/リンク/タグ/「続きを読む」は個別動作を維持＝stopPropagation・写真ボタンはキーボード/SR の主導線として残す・#101）。**読み込み中はカード形のガラス pulse スケルトン**（FeedSkeleton・素っ気ない「読み込み中…」を置き換え・#99）。カード（写真）の角丸は控えめ（rounded-xl・#100）。**大画面（lg）はカード幅を広げ（max-w-5xl）、余白は主に写真を大きく**（写真 224→288px・#56）。植物タグで絞る。
- **投稿頻度の高い人をフィードで「減らす」（#138）**: 人ごとに **1/2・1/5・1/10** の間引き度合いを設定でき、その人の投稿をフィードで決定的に間引いて表示する（ミュートの手前の柔らかい手段）。判定は `hash(post.id) % level === 0`（FNV-1a・id 由来）＝**deterministic**でリロードしても残る投稿は同じ・相対順序は保つ。状態（pubkey→level）は localStorage（`src/lib/feed/dilution.ts`・SSR 安全 getLS・純関数 `diluteFeed(posts, map)`）。適用は取得済みリスト→表示の間（`PostGrid`＝フィード/discover 共通）に1段挟む。**`/me`（自分の投稿）には掛けない**＝自分を減らさない。UI は投稿詳細モーダルの著者ヘッダ下に置く `DilutionControl`。**既定は畳む**（控えめな入口・未設定は中立文言「{名前}さんの表示を調整」／設定中は「1/N に減らし中」）＝既定が「なし」なのに常駐すると「減らすのが基本」と誤解させるため。開いたときだけ **なし・1/2・1/5・1/10 の4点にスナップするスライダ**（native range＋`aria-valuetext`・中間の無意味値を出さない）が出る。見出しは**人に明示**「{名前}さんの投稿をフィードで減らす」（"薄める"は植物アプリで"植物を薄める"とも読めるため廃止）。設定すると即フィードに反映（同タブはカスタムイベント、別タブは storage で同期）。
- **トップ と みんなの植物 の住み分け（#52）**: **トップ**＝`t:hanoba` のみ（葉の場＝ローカル）。**みんなの植物（discover）既定表示**＝`#plantstr`（Nostr 全体の植物界隈）∪ `t:hanoba` の**マージ**（id 重複除去・時系列）。hanoba は全投稿に `t:plantstr` を自動付与する（#383・reciprocity＝§6。discover が `#plantstr` を読むだけの非対称を解消し、自分の投稿を Nostr 全体の植物界隈にも露出させる）。`t:hanoba` 経由でも自分の投稿が「みんな」に出る（「投稿したのに見えない」混乱の解消）。個別検索（`?q=`）は §6 の横断検索のみ。各画面に1行の住み分け説明を置く。
- **人気ランキング（#162・`/ranking`）**: 植物**品種**の人気を音楽チャート風に出す。**バックエンドレス**（§6・集計サーバを持たない）で、取得済みの **`t:hanoba` 投稿だけ**（`fetchRankingPosts`・既定 limit 500・画像ありのみ＝フィードと同母集団）を**クライアントで数える**「読むたびに数える」方式。母集団を `t:hanoba` 限定にするのは、**自分の投稿が見えてチャートを動かす＝投稿の動機**にするため（discover の横断集約は混ぜない）。集計と差分は純粋ロジック `src/lib/feed/ranking.ts`（relay も `Date.now()` も持たず `now` を引数で受ける＝テスト可能）に集約し、島 `RankingBoard` は取得・`now=Math.floor(Date.now()/1000)` の供給・描画だけを担う。
  - **集計単位＝品種**: 各投稿の `hashtags` を **`buildFuda`（§4 札・#182/#23）** にかけて品種（`{key,name(和名),sci(学名)}`）に畳み、品種ごとに投稿数を数える（`tallyVarieties`）。属/品種の畳み込み・alias の canonical 寄せ・**同一投稿内の重複 dedupe** は `buildFuda` に一本化する（同じ品種の別名を1投稿で複数付けても1票）。同一品種が複数投稿に出れば投稿ごとに加算。票数降順・同数は**初出順**で安定。

  - **先週比（`rankWithDeltas`）**: 「直前の週」＝現在週より前で**データのある直近の過去週**（カレンダー連続でなくてよい）。**NEW**＝現在週より前のどの週にも一度も載っていない。**RE**＝過去に載ったが直前週には不在で今週復帰。**↑N/↓N/—（same）**＝直前週との順位差。**直前の過去週が無い（初週）なら全行 NEW**（偽の矢印を出さない）。NEW/RE は**取得済みの直近ウィンドウ内**（`fetchRankingPosts` 上限＝現状およそ直近 500 投稿）での判定で、全履歴に対する絶対判定ではない。**初週バナー**（「先週比は来週から」）は全行 NEW を代理にせず**データのある週数が実質 1 週以下**のときだけ出す（後発の週がたまたま新規品種ばかりでも誤って初週と出さない）。

- **学名表記（#70）**: 学名は属名・種小名をイタリックにするのが正式だが、ランク略号・接続語（`var.`/`subsp.`/`ssp.`/`f.`/`cv.`/`aff.`/`cf.`/交配の `×` 等）は**直立体**にするのが植物学の正しい表記。`src/components/ui/SciName.tsx` が sci 文字列をトークン分割し接続語だけ直立で描画する。PostDetail の植物チップで使う。
- **学名データの丸め方針（#23/#182・`variety-catalog.ts` の `sci`）**: 荒い界隈の表記（Spor/PUP・自分で名乗る品種・属止まり）をどこまで正規化するかの規約。
  1. **書式＝ICNCP**: 属名大文字始まり・種小名小文字（`Pachypodium brevicaule`）。**cultivar は `Genus 'Cultivar'`**＝シングルクォート囲み・各語頭大文字・ただし冠詞/前置詞/接続詞は言語慣習で小文字（`Pierre de Ronsard`・`Perle von Nürnberg`・`Black and Blue`）。**ラテン文字に統一**（漢字 cultivar＝富貴蘭/長生蘭/さつき 等の古典銘も読みを裏取りして romaji 化＝漢字は和名 side で見せる）。
  2. **種が特定できない属通称** → `Genus sp.`（属は当て種は未特定。例 `Sansevieria sp.`・「その他塊根」の通称）。
  3. **状態語・栽培アクション**（実生/発芽/発根/仔株・Spor/PUP・spore 等）は**品種ではない**＝学名を持たせない。これらは `variety-catalog` でなく **`tag-catalog.ts` の世話/記録タグ**に置く（識別子＝品種 と 状態 は別軸＝タグ/札の分離と同じ筋）。
  4. **登録の無い趣味家銘（勝手に名乗る品種）** → 確実な学名が無ければ `Genus '通称ローマ字'` に丸める（白鯨→`Agave titanota 'Hakugei'`）。正式登録名でないものは要検証 follow-up（#182）。
  5. **グループ概念**（コケ/シダ/高山植物/多年草 等＝species でない総称）は `sci` を持たない＝札は**和名のみ**（グレースフル）。
  6. 中身（綴り・読み・trade designation と cultivar の区別）の精度は #23「育てる辞書」として実運用で上げる。書式は上記で統一済み。
- **ヘッダ（全ページ共通・#30）**: 主要ナビ（みんなの植物／自分の植物／Hanōba とは／投稿する）を `SiteHeader.astro` に内蔵。sm 以上はインライン、sm 未満はハンバーガー→ドロップダウン。ページごとの `current` で現在地を強調。`hidden sm:inline` でモバイル時にリンクが消える旧構成は廃止。

## 5. デザインシステム ── 「夜の温室／大人のコレクター部屋」

### 5.0 世界観（確定・kako-jun session640）

Hanoba は明るい量販 SNS ではない。**焦げ茶の木目棚に植物が並び、オレンジの照明がぼうっと灯り、ガラスには水滴が伝う、暗くシックで高級感のある夜の部屋**。プロの植物コレクターの空間。ここに「育てて、見せる。」がある。Instagram の白い量産グリッドの対極＝大人の落ち着きと所有の喜び。

実装3原則:
1. **背景はベタ塗りにしない** ── AI 生成のぼやけた高級写真（`public/og/bg-blur.webp`）を `background-attachment: fixed` で全面に敷き、暗いオーバーレイで可読性を確保する。
2. **面（パネル）は半透明のグラス** ── `.glass` / `.glass-strong`（`backdrop-blur`＋微光の白枠）。ガラス越し・水滴・反射の空気を出す。
3. **ボタンだけは塗りでくっきり** ── CTA は `bg-ha-green`（葉）／`bg-ha-pink`（花）の塗りで可読性最優先。グラスに埋もれさせない。

### 5.2 タイポ・モーション

- **タイポ**: 見出し＝`Outfit`（geometric sans・欧/数字）×日本語 `Zen Maru Gothic`（丸ゴシック・角丸の世界観に合わせる #170）。本文＝Zen Maru Gothic。本文400↔見出し800/900でウェイト飛躍。汎用フォント（Inter/Roboto/system/Space Grotesk）禁止（Anthropic frontend aesthetics 準拠）。`--font-display` / `--font-sans`。
- **モーション**: ページロードの控えめな staggered reveal（`.ha-rise`＋`--i`・CSS-only・`prefers-reduced-motion` で無効）。**安っぽい装飾は禁止**（ホバーでロゴを傾ける等はダサいので不可）。
- **画像リビール（#145）**: 画像は blur-up リビール（`ProgressiveImage`＋`.ha-reveal`）で出す。デコード完了まで `opacity:0` で隠し、`onLoad`（＋キャッシュ済みは `img.complete`）で `data-loaded="true"` を立てて blur＋わずかな scale を解きながらフェードインする。非プログレッシブ画像が上から帯状に描かれる生 `<img>` を置き換えたもの。`prefers-reduced-motion` では遷移なしで即時表示。4 箇所（カード写真＝object-cover 固定箱／拡大カルーセル＝object-contain／アバター＝rounded-full／あなたの植物グリッドのサムネ＝object-cover 正方形）で共有。WASM/Canvas の「現像」リビール（stage 3）は別途の将来フォローアップ。
- **フローティング UI**: 右下に「一番上へ戻る」（`ScrollToTop`・全ページ共通・400px 超で出現・`prefers-reduced-motion` 時は smooth でなく即時・#110）。**投稿 FAB（#283）**: その左隣（`PostFAB`・`right-[4.75rem]`＝隅は ScrollToTop 用に空ける固定スロット）に常駐の塗り緑 FAB（`/compose` へ1クリック・**アイコンは「横から見た綿毛」＝写真を風に放つメタファ**〔`public/post-fab.webp`・gpt-image-2 生成の白透過ラスタ。真上ビューの放射状＝＊状に紛れる `sprout`/`dandelion` を避けるための、線アイコン集 `Icon.tsx`(currentColor SVG) の例外〕・ScrollToTop の表示状態に依らず位置不動）。投稿画面（`/compose`）では `MainLayout` の `isCompose` でサーバ側に出さない。
- **綿毛の送信エフェクト（#148 / #252）**: 投稿ボタンを押すと綿毛（タンポポの種）が舞い上がる。打ち上げの一斉バーストに加え、**投稿が終わるまで少量ずつ連続スポーン**して「風に乗って次々舞い上がる」流れを作り、アップロード＋publish の長い待ち（10秒級）でも画面が止まって見えないようにする（#252）。メタファは「自分の植物の写真を風に放つ」。粒は和・水彩タッチの**透過スプライト3変種**（`public/seed-watercolor-{1..3}.png`＝黒い綿毛を白地に生成し、輝度をアルファ＋白黒反転して白い綿毛＋半透明の冠毛に変換＝クロマキーのハロ無し・API キー不要）をランダムに割り当て、大きめサイズ（~104px）＋**粒ごとの非一様スケール/skew の変形**（見かけの拡大率を混ぜる）＋横揺れ・回転ドリフト・上端フェードで一粒ずつ違う舞いにする（反復＝単調を消す）。種の生成は純関数 `lib/composer/dandelion.ts`（`makeWind`/`makeSeed`/`makeSeeds`・連続側はバッチごとに風を取り直す）、描画は `DandelionBurst`（`active={posting}` で駆動・各粒は animationend で自分を消す）＋`.ha-seed-rise`。**ボタンに残す稼働サインは綿でなく普通のスピナーリング**＋段階テキスト「写真を送信中 N/M → 投稿中」（綿はボタンから飛び出した側なので・#252）。idle の送信ボタンのアイコンは**投稿 FAB と同じ綿毛ラスタ `public/post-fab.webp`** に統一（#293・線アイコン `dandelion` から差し替え＝右下 FAB と同じ絵）。`prefers-reduced-motion` では spawn せず無音（コンポーネントが null・CSS でも `animation:none`）。`pointer-events:none`・`aria-hidden` でクリック・レイアウト・支援技術に干渉しない。**綿毛が舞ってもスクロールバーの出没で画面幅がガタつかないよう、`html` に `scrollbar-gutter: stable` で gutter を常時確保する（#294・横方向は `body{overflow-x:clip}` で切る #271 の積み残し対応）。**
- **天気の環境演出（#231・#132 段階4）**: ハノーバ（＝金沢駅・鼓門の固定座標）の今の天気を Open-Meteo から取得し（ユーザー位置は使わない＝身バレ回避・backendless）、雨のとき**ガラスに付いた水滴**を前面環境レイヤ（`AmbientWeather`・`client:idle`）で重ねる。**「天気＝明るさ」でなく「天気＝ガラス面に起きる現象」**で表す＝明度は上げずに暗色シック（§5.0）を保つ（晴れでも眩しくしない＝「背景が明るくなると暗色前提が崩れる」矛盾の根本解決）。レイヤは**部屋写真と本文の間（`z-index:-1`）に固定**＝本文の可読性は無傷・`pointer-events:none`。雨の強度（WMO `weather_code`＋`precipitation`）で素材を出し分ける（**大雨＝密な水滴 `rain-glass-heavy.webp`／普通・霧雨＝疎な玉＋流れ落ちる雫 `rain-glass.webp`**）。`opacity` は「ほどほど」（0.65）で滲み出し＋ごく緩い呼吸（drift）。`prefers-reduced-motion` で静止。`?weather=rain|heavy|drizzle|clear…` で強制できる（dev/blink 用）。純ロジックは `src/lib/weather/`（URL 生成・WMO→状態・強度・キャッシュ・取得＝`now` を引数で受けテスト可能）、島は取得と描画だけ。**段階2 は雨のみ**＝背景写真の天気別差し替え・昼夜・lore テキスト連動は後段。
- **時刻の環境演出（#231 後段②）**: **背景写真は差し替えず**、同じ絵にかけるオーバーレイ（body の `background-image` の `linear-gradient`）の**暗さ・色温度だけ**を時間帯で変調する。軸は**鼓門の JST**＝天気と同じく「みんな同じハノーバの今」を共有（ユーザー TZ に依らない）。朝/昼/夕/夜の 4 段（`timeOfDay(jstHour)`・`src/lib/weather/timeOfDay.ts`・純関数）。**全段ダークレンジ内**＝暗色シック維持・「天気＝水滴／時刻＝明るさ／季節＝背景写真そのもの」の3軸分離。**夜＝暖色で最暗（既定・これまでの見た目）／夕＝鮮やかな夕焼けオレンジ（一番"ばえる"）／昼＝中立やや冷で最も明るい（室内の緑が見える程度・眩しくしない）／朝＝冷たい薄明**。実装＝`:root` の `--bg-overlay-{top,mid,bot}` を `html[data-time="…"]` で上書き。**flash 回避**に MainLayout の `<head>` inline script が**描画前に** `data-time` を設定（バケットは `timeOfDay.ts` と同じ境界を直書き＝市民手帳ラベル #262 と同じ手法）、`AmbientCalendar` 島がマウント後に同値で再設定＋5 分ごとに境界更新。`?time=morning|day|evening|night` で強制（dev/blink 用）。天気の水滴とは独立に合成される（夕焼け＋雨 等）。
- **季節の環境演出（#231 後段①）**: **背景写真そのもの**を八節（二十四節気の主要8つ＝四立＋二分二至）で差し替える＝「同じ夜の collector 部屋がゆっくり移ろう」。軸は**鼓門の JST 暦日**（`solarPhase(month,day)`・`src/lib/weather/solarPhase.ts`・純関数・節の開始は近似固定日／背景の気分なので天文精度不要）。**室内・夜・暗いは全節共通**＝黒格子の壁・板付ビカクシダ・アガベ・ユッカの部屋は据え置き（屋外絵は使わない）、**塊根の状態（落葉性＝冬は裸の幹／夏は繁茂・開花）＋季節の一鉢（立春=梅／春分=桜／立夏=紫陽花／夏至=朝顔／立秋=ススキ／秋分=紅葉／立冬=南天／冬至=松・正月）**で節を表す（常緑のアガベ等は不変＝世界観を壊さない）。アングルは節ごとに変えて単調を避ける。実装＝背景写真を `--bg-photo` 変数にし `html[data-sekki="…"]` で8枚（`public/weather/season-*.webp`）を差し替え（時刻のオーバーレイ・天気の水滴はその上に重なる＝3軸合成）。未知の節は既定 `bg-blur.webp` にフォールバック。**flash 回避**に inline script が描画前に `data-sekki` を設定（境界は `solarPhase.ts` と同じ値を直書き）、`AmbientCalendar` 島が再設定＋更新。`?sekki=risshun|shunbun|rikka|geshi|risshu|shubun|ritto|toji` で強制（dev/blink 用）。
- **グリッド**: 写真は Instagram explore 流に**隙間を詰め（gap 0.5）角丸は小さく（rounded-md）フラット**（浮かせない）。

### 5.3 ワードマーク／アイコン

### 5.6 操作（ボタン/リンク）の配置 — 統一ポリシー（#98）

## 6. データ（Nostr）— 検証済み契約

mypace と同じ Nostr 空間・同じタグ規約に準拠する（独自化しない＝mypace から見える要件）。

- **イベント**: `kind: 1`（テキストノート）。
- **タグ**:
  - `['t','mypace']` … **必須・自動付与**。これで投稿は通常の mypace 投稿と同一に mypace タイムラインへ出現。
  - `['t','hanoba']` … **必須・自動付与**。hanoba のフィードはこのタグで絞る（mypace の一般投稿を hanoba に流さないため）。
  - `['t','plantstr']` … **自動付与（#383・mypace 完全一致からの意図的逸脱）**。Nostr 全体の植物界隈（`#plantstr`）へ hanoba の投稿も露出させ、discover が `#plantstr` を「読む」のに「書かない」非対称を解消する（reciprocity・生態系への還元）。**kill switch**＝`buildAutoTags()` のこの1行を消して再デプロイすれば全ユーザー分が止まる（可逆）。**これはコミュニティ・ハッシュタグの自動付与であって、本文 `#` を `t` タグ化するルールではない**（下記の「本文 `#` は `t` 化しない」とは別軸）。
  - `['client','hanoba']` … 自動付与。可視性には影響しない。
  - **植物トピック・種別は、一言（本文）にユーザーが書く `#ハッシュタグ`** で表現（例: `#アガベ #パキポ`）。本文に見えるテキストとして残す。**hanoba 側に固定の植物 taxonomy は持たない**。**本文 `#` は `t` タグ化しない**（mypace と完全一致＝独自化禁止。確定）。集約は書き込み側のタグでなく、読み取り側の二段構え検索で行う（下記）。
- **content**: 必須の一言 ＋ 画像 URL（1〜4個）を**インラインのプレーン URL**で（`https://image.nostr.build/<hash>.<ext>`。imeta タグは使わない）。**一言が空（URL のみ）は弾く**。**本文 caption 末尾に `#plantstr` も自動併記する（#408・#383 follow-up）**＝`t:plantstr` タグに加え本文文字列としても `#plantstr` を残し、① NIP-50 全文検索（本文の "#plantstr" を引く層）／② 本文 `#` を parse する他クライアント の双方へ hanoba 投稿を届かせ reach を最大化する（`buildNoteTemplate` の content 合成・画像 URL より前＝caption 側）。**重複ガード**＝caption に既に `#plantstr`（大小無視）があれば足さない（手書きを二重化しない・`extractHashtags` で判定）。**kill switch**＝`buildNoteTemplate` のこの併記数行を消せば止まる（可逆）。これは `t:plantstr` 自動付与（コミュニティ・ハッシュタグ）の本文側補完であって、**品種など本文 `#` を `t` 化するルールではない**（その別軸は不変）。
- **画像アップロード**: `POST https://nostr.build/api/v2/upload/files`（NIP-98 認証）。EXIF はサーバ側で自動削除。
- **リレー**: `wss://relay.damus.io`, `wss://nos.lol`（必要時に検索 `wss://search.nos.today`）。
- **鍵**: ブラウザ生成・localStorage 保存、NIP-07 任意。

- **タグの集約（読み取り）＝二段構え**（mypace 準拠・#3/#4 で実装）: ① `{"#t":[tag]}` フィルタで `t` タグ持ちを取得、② NIP-50 `search:"#tag"` で**本文ハッシュタグ**を全文検索（他クライアントの `t` タグ無し投稿も拾う）、③ event id でマージ・重複除去。これが emergent taxonomy の本体で、書き込み側で本文 `#` を `t` 化しなくても集約できる根拠（書き込みタグは `t:mypace` / `t:hanoba` / `t:plantstr` / `client:hanoba`＝うち `t:plantstr` は #383 の reciprocity 自動付与で、本文 `#` の `t` 化ではない）。検索 relay = `wss://search.nos.today`。

- **植物の別名 OR 検索（#23 Phase 2・実装済）**: タグ軸が辞書の植物に解決できる場合（`tagAliasValues`＝`findPlantByTerm`/`plantTagValues`）、その植物の**全表記を横断**する。多軸エンジン `fetchDiscoverFiltered`（#131）が `#t` を別名展開して OR 取得（`{"#t":[名前, 別名, 単語学名…]}`）し、`applyClientFilter` の軸間 AND もタグごとに**別名集合で照合**する（例: タグ「パキポ」が `#グラキリス` 投稿に一致）。PostDetail の植物チップはそのタグに絞り込み（`#<著名名>`→tags 軸）この経路に乗る。**別名集合は dictionary（#23）∪ `variety-catalog` の属/品種別名（#303・札と同じ source of truth）**＝辞書外の属別名でタグした cross-client 投稿（例 `#ゴムの木`＝フィカスの別名・dictionary に無いが catalog にある）にも AND が当たる（`buildCatalogAliasIndex`＝filtering 時に variety-catalog を動的 import して索引を1回構築・キャッシュ、`tagAliasValues` と合成。失敗時は dictionary だけで動くグレースフル）。hanoba native 投稿は canonical 属名を書くので元から往復し、本件は cross-client/手打ち非 canonical の recall 改善。
- **コメント＝Nostr リプライ（#142・実装済）**: 投稿へのコメントを **NIP-10 のリプライ（`kind:1`）** として表現する（投稿詳細モーダルの `CommentSection`）。これは表示専用だったいいね（kind:7 集計）に対する**書き込み側の相棒**＝hanoba から Nostr に「足す」もう一つの経路。
  - **タグ**: `["e", <親投稿id>, "", "root"]`（親投稿を root に印付ける direct reply。relay ヒントは空）＋ `["t","mypace"]` ＋ `["client","hanoba"]`。
  - **`p` タグ（@呼びかけ）は付けない**: コメントは投稿者を名指しで呼び出す通知でなく「その投稿にそっと添える一言」（静かな観賞 SNS の posture・確定）。
  - **`t:hanoba` は付けない**: hanoba フィード/人気タグ集計は `#t:hanoba` で絞って**画像投稿だけ**を集める（`fetchHanobaFeed` / `fetchPopularHashtags`）。文章だけのコメントに `t:hanoba` を付けるとフィードのノイズ・タグ集計の汚染になる。コメントは親の `e` タグ経由（`{kinds:[1], "#e":[postId]}`＝`fetchReplies`）で読むので不要。
  - **読み取り**: `{kinds:[1], "#e":[postId]}`（`fetchReplies`）。このフィルタは親を `e` タグで参照する kind:1 を**全部**返すので、引用リポスト（NIP-18・`["e", postId, "", "mention"]`）が混ざる。`toComments(events, postId)` は**本物のリプライだけを残す**＝「値が `postId` に一致し marker（`tag[3]`）が `"mention"` でない `e` タグ」を1つでも持つもの（`"root"`/`"reply"`/マーカー無し＝残す、`"mention"`＝引用リポストは落とす）。あわせてリレー間の重複は id で除去。表示は古い順/新しい順を切り替え（`sortComments`）。**削除**は投稿と同じく NIP-09 `kind:5`（`deleteComment`。コメントは画像を持たないので実体削除は無い）。
  - ビルダーは `buildReplyTemplate`（純関数・空 content / 空 parentId は throw）、relay 呼び出しは `client.ts`（`fetchReplies` / `publishReply` / `deleteComment`）に集約（§3）。本文 `#` を `t` 化しない §6 契約は不変。
- **住民投票＝Nostalgic BBS 埋め込み（#160・`/vote`）**: 市民手帳の街の地図ページ（#163・#469）末尾の「市政の窓口」strip に置く、最初に開庁した役所（ヘッダ/フッタに無く手帳が唯一の入口なので、図鑑化で手帳のメニューを外した際もここへ退避＝孤立させない）。投票（意見集約）は **kako-jun 自作のバックエンドレス掲示板 Nostalgic BBS**（外部ホスト・`https://nostalgic.llll-ll.com`）を web component で埋める＝hanoba 側にサーバ/DB を足さない（§7 non-goal 維持）。
